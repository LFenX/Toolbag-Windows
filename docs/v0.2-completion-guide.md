# Toolbag v0.2 收尾指南

这份清单只处理 v0.2 框架重构之后还没有闭环的事：签名信任链、Registry 分发链、第一个外部插件、最终发布。推荐按顺序做，不要一上来就同时开很多战线。

## 0. 先冻结主仓状态

目标：确认当前重构可以作为 v0.2 的基线。

1. 在主仓跑完整检查：

   ```powershell
   corepack pnpm check
   Push-Location src-tauri
   cargo fmt --check
   cargo clippy --all-targets -- -D warnings
   cargo test --lib
   Pop-Location
   ```

2. 启动一次本地应用：

   ```powershell
   corepack pnpm tauri:dev
   ```

3. 手动验收这几条：
   - `/marketplace` 可以打开，没网时显示 bundled/cached 空态。
   - `/plugins` 能看到内置 `com.lfen.toolbag.environment-overview`。
   - 关于页能看到数据目录和签名状态。
   - 状态栏显示版本、签名状态、在线状态。

4. 没问题后提交主仓重构。建议提交信息：

   ```text
   feat: add v0.2 plugin runtime foundation
   ```

## 1. 分清两套密钥

v0.2 需要两套密钥，作用不同，不能混用。

| 密钥 | 用途 | GitHub Secret |
|---|---|---|
| Tauri updater key | 给应用本体更新包签名 | `TAURI_SIGNING_PRIVATE_KEY` / `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` / `TAURI_UPDATER_PUBKEY` |
| Toolbag plugin minisign key | 给 `.tbpkg` 插件包签名 | `TOOLBAG_PLUGIN_PUBKEY`，私钥给插件发布 workflow 使用 |

应用本体 release 构建会把 `TOOLBAG_PLUGIN_PUBKEY` 编译进二进制。没有这个值，release workflow 应该失败，因为 release 版必须强制验插件签名。

## 2. 生成 Tauri updater 密钥

如果以前已经配置过应用更新密钥，可以跳过本节。

1. 本机生成：

   ```powershell
   corepack pnpm tauri signer generate -- -w "$env:USERPROFILE\.tauri\toolbag.key"
   ```

2. GitHub 主仓 `LFenX/Toolbag-Windows` 设置：
   - `TAURI_SIGNING_PRIVATE_KEY`：私钥文件内容。
   - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`：生成时设置的密码；没有就留空。
   - `TAURI_UPDATER_PUBKEY`：命令输出的 updater 公钥。

3. 可选 Repository Variable：
   - `TOOLBAG_UPDATER_ENDPOINT`
   - 默认可以先用 `https://github.com/LFenX/Toolbag-Windows/releases/latest/download/latest.json`。

## 3. 生成插件 minisign 主密钥

目标：以后所有官方插件包都用这把 key 签名，Toolbag release 版只信任它。

1. 安装 minisign 后生成密钥：

   ```powershell
   New-Item -ItemType Directory -Force "$env:USERPROFILE\.toolbag-keys"
   minisign -G -p "$env:USERPROFILE\.toolbag-keys\toolbag-plugin.pub" -s "$env:USERPROFILE\.toolbag-keys\toolbag-plugin.sec"
   ```

2. 备份私钥：
   - `toolbag-plugin.sec` 放 1Password 或离线介质。
   - 不要提交到任何仓库。

3. 主仓 GitHub Secret：
   - `TOOLBAG_PLUGIN_PUBKEY`：建议填 `toolbag-plugin.pub` 的完整内容。
   - 当前代码也接受单行 `RW...` 公钥，但完整文件内容更不容易搞错。

4. 后续插件仓需要能签包，建议使用另一个 Secret：
   - `TOOLBAG_PLUGIN_SECRET_KEY`：`toolbag-plugin.sec` 的完整内容。
   - 只给官方插件仓或受信任发布仓配置。

## 4. 搭建 `LFenX/Toolbag-Registry`

目标：让客户端能从 `https://toolbag.lfen.dev/index.json` 或 GitHub Pages fallback 读到插件索引。

1. 新建公开仓库：`LFenX/Toolbag-Registry`。

2. 仓库结构：

   ```text
   plugins/
   schemas/
   scripts/
   .github/workflows/
   categories.json
   app-versions.json
   README.md
   ```

3. 先写 `categories.json`：

   ```json
   [
     { "key": "系统", "label": "系统", "icon": "monitor", "order": 1 },
     { "key": "网络", "label": "网络", "icon": "globe", "order": 2 },
     { "key": "开发", "label": "开发", "icon": "code", "order": 3 },
     { "key": "实用", "label": "实用", "icon": "wrench", "order": 4 },
     { "key": "媒体", "label": "媒体", "icon": "hard-drive", "order": 5 },
     { "key": "安全", "label": "安全", "icon": "shield-check", "order": 6 },
     { "key": "其他", "label": "其他", "icon": "boxes", "order": 7 }
   ]
   ```

4. 写 `app-versions.json`：

   ```json
   { "stable": "0.2.0", "beta": null }
   ```

5. 写 `scripts/build-index.mjs`：读取 `plugins/*.json`，取每个插件最新 release，生成 `public/index.json`，并复制 `categories.json`、`app-versions.json`、`schemas/`。

6. 配 GitHub Pages：
   - Source 选择 GitHub Actions。
   - workflow 在 main 分支变更 `plugins/**`、`categories.json`、`app-versions.json` 时构建。

7. 先部署一个空 index，确认能访问：
   - `https://lfenx.github.io/Toolbag-Registry/index.json`
   - 自定义域名准备好后再切 `https://toolbag.lfen.dev/index.json`。

## 5. 做第一个外部插件

建议先做 `com.lfen.toolbag.hash`，因为它能覆盖文件读取、文本输入、进度、结果渲染，但风险低。

1. 新建插件仓：`LFenX/toolbag-plugin-hash`。

2. 最小结构：

   ```text
   tool.json
   ui.json
   icon.svg
   README.md
   src/
   .github/workflows/release.yml
   ```

3. 首版命令建议两个：
   - `hash.text`：无权限，直接对文本算 hash。
   - `hash.file`：声明 `fs.read`，对用户选择文件算 hash。

4. 本地打包产物必须有三件：

   ```text
   toolbag-plugin-com.lfen.toolbag.hash-1.0.0.tbpkg
   toolbag-plugin-com.lfen.toolbag.hash-1.0.0.tbpkg.sig
   toolbag-plugin-com.lfen.toolbag.hash-1.0.0.tbpkg.sha256
   ```

5. 发布到 GitHub Release 后，在 Registry 仓添加：

   ```text
   plugins/com.lfen.toolbag.hash.json
   ```

6. 合并 Registry PR，等 Pages 更新后，在 Toolbag `/marketplace` 安装它。

## 6. 端到端验收

按这个顺序验，哪里失败就停在那里修。

1. 主仓 release workflow 能启动，并且缺少 `TOOLBAG_PLUGIN_PUBKEY` 时会失败。
2. 配好 `TOOLBAG_PLUGIN_PUBKEY` 后，release workflow 能产出 NSIS 和 updater artifacts。
3. Registry 空 index 能被 `/marketplace` 拉到，来源显示 live。
4. 示例插件 release 产物的 sha256 与 Registry 中一致。
5. 示例插件 `.sig` 能通过主仓编译进来的公钥校验。
6. `/marketplace` 安装插件成功。
7. `/plugins` 可以禁用、启用、卸载示例插件。
8. `hash.text` 和 `hash.file` 都能跑通，日志、进度、结果渲染正常。
9. 删除网络后重启应用，Registry cache/bundled fallback 行为正常。

## 7. 发布 v0.2.0

1. 确认 `package.json`、`src-tauri/Cargo.toml`、`tauri.conf.json` 版本都是 `0.2.0`。
2. 更新 `README.md`、`docs/release.md`、`docs/adding-tool.md` 中的最终 URL。
3. 打 tag：

   ```powershell
   git tag v0.2.0
   git push origin v0.2.0
   ```

4. GitHub Release 是 draft，先下载安装包到干净 Windows 环境试装。
5. 试装通过后发布 Release。

## 8. 当前优先级

建议现在只做这三件：

1. 提交主仓 v0.2 重构。
2. 配好两套签名密钥，跑一次 release workflow。
3. 搭空 Registry，让 `/marketplace` 先拿到 live index。

示例插件等上面三件稳了再做。那时候问题会更少，调试面也会小很多。
